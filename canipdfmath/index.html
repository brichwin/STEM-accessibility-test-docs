PDF/UA-2 Math Encoding Test Results

Test results for PDF/UA-2 documents containing math encoded in combinations of three formats: Associated Files (AF), Structure Elements (SE), and Alt text on the formula tag. Each combination was tested in Adobe Acrobat Pro and Firefox with NVDA and JAWS screen readers.

Encoding formats tested:

  • AF: MathML stored as an Associated File attached to the formula object
  • SE: MathML embedded directly in the PDF tag tree as Structure Elements
  • Alt text: Auto-generated plain text alternative on the formula tag (Alt attribute) using \tagpdfsetup{math/alt/use}

Understanding the results

The goal of accessible math in a PDF is not simply to have something read aloud. It is to give screen reader users the same depth of access that sighted readers get when they look at a formula. That means being able to hear math spoken in natural, accurate, and unambiguous mathematical language; navigate into an expression interactively; and optionally receive braille output on a refreshable braille display.

This is what Configurable math speech/braille output; interactive expression exploring means in the results below. It is the only outcome marked as passing (green) because it is the only outcome where the MathML in the PDF is being processed by a dedicated math accessibility engine (like MathCAT, which is now built natively into both NVDA and JAWS) rather than being ignored, mangled, or substituted with an inferior plain text fallback.

The other outcomes represent meaningful failures:

  • Alt text: the screen reader reads a static plain-text string instead of the MathML. Even assuming it is acurate, it's likely not in the user's preferred math speech style, there is no interactivity, no braille output, and no way to navigate the expression. For simple inline values this may be tolerable, but for complex expressions it is a significant loss of information and control.
  • Speaks raw MathML: the screen reader reads the underlying XML markup aloud verbatim, which is unintelligible to listeners and unusable in practice.
  • Math speech character by character: the reader encounters the MathML tag tree as individual characters rather than as a parsed mathematical expression, producing output that is technically present but semantically meaningless.
  • Math is skipped over; not read aloud: the formula is silently passed over with no output whatsoever. The reader has no indication that math was present.
  • Speaks as gibberish: the reader produces output, but it does not correspond to the mathematical content in any meaningful way.
